## 의사결정
### gpp 스케줄러 
도입 이유 : 모집상태 변경을 위해 도입, 원하는 동작 시점(트리거)을 설정

#### 예상 질문
- 상태값 컬럼 변경 자동화를 위해 도입했는데, 다른 기술은 뭐가 있는지 알아보았는지?

네, 스케줄러 외에도 몇 가지 대안을 검토했습니다.
대표적으로는:
DB Event Scheduler (MySQL EVENT)
메시지 큐 기반 지연 처리 (Kafka, RabbitMQ Delay Queue)
배치 프레임워크 (Spring Batch)
애플리케이션 이벤트 기반 처리

각각 비교해봤을 때,
이번 요구사항은 정해진 시간 기준으로 상태를 주기적으로 갱신하는 작업이었기 때문에
구조 복잡도가 낮고 운영이 쉬운 Spring Scheduler가 가장 적합하다고 판단했습니다.

특히:
- 실시간 이벤트 기반 아님
- 분 단위 주기 처리면 충분
- 별도 인프라 추가 부담 없음
이 점에서 스케줄러를 선택했습니다.

- 벌크 업데이트(일괄 반영)를 사용한 이유는?
모집 상태 변경 대상이 다수 발생할 수 있어, 
엔티티를 하나씩 조회 후 업데이트하면 다음 문제가 있다고 판단했습니다.

- N건 SELECT + N건 UPDATE 발생
- 영속성 컨텍스트 오버헤드 증가
- 대량 데이터에서 성능 저하

그래서 조건 기반으로 한 번에 처리하는 벌크 업데이트를 사용했습니다.

이를 통해:
- DB round-trip 최소화
- 대량 데이터 처리 성능 개선
- 스케줄러 실행 시간 단축

효과를 얻었습니다.

## 성능 개선
### gpp 조회수
Redis 도입 이유 : 데이터 쓰기-캐싱 후 일괄 반영을 위해 도입, DB비용 및 부하 감소

#### 예상 질문
- Redis를 왜 도입했나요?
조회수는 쓰기 빈도가 매우 높은 데이터 특성이 있어,
매 요청마다 DB에 직접 반영하면:
- DB write 부하 증가
- lock 발생 가능
- 응답 지연 발생
문제가 있다고 판단했습니다.

그래서 Redis에 먼저 누적 후
주기적으로 DB에 반영하는 Write-Behind(쓰기 캐싱) 전략을 적용했습니다.

이를 통해:
- DB write 트래픽 감소
- 응답 속도 개선
- DB lock 경합 완화

효과를 얻었습니다.

- 조회수에 쓰기 캐싱을 선택한 이유는?
조회수의 병목은 읽기보다 쓰기 빈도에 있다고 판단했습니다.
현재 서비스에서 중요한건 정확한 조회수값의 즉각적 반영이 아니라, 
실제 값의 반영이 늦더라도 효율적인 쓰기 처리를 하는것이 
서비스 DB의 부담을 줄여주는 안정적 운영에 도움이 된다고 생각했습니다.
