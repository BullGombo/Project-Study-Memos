비밀번호를 Body로 받는 방식은 이대로 괜찮은지?
DELETE 메서드에서 RequestBody로 비밀번호를 받는 방식의 단점
1. DELETE 요청은 RequestBody 사용을 권장하지 않는다 (HTTP 표준 측면)
2. 비밀번호가 API 로그에 남을 위험이 있다
3. RESTful 하지 않음 (일반적 DELETE 패턴과 다름)
REST에서 리소스 삭제는 보통 다음 중 하나:
인증된 사용자(JWT 등)가 호출할 수 있는 방식
DELETE /users/me
비밀번호는 인증 과정(JWT)에 의해 이미 검증됨
즉, 삭제 요청 안에서 비밀번호 검증을 또 하는 패턴 자체가 REST스럽지 않다.
4. 안전한 HTTP 캐싱 정책과 충돌 가능
5. 보안적으로 취약 — HTTPS를 사용해도 전송 중 평문 구조
6. 클라이언트(프론트엔드)에서 DELETE + Body 보내기 불편
