HTTP의 특성

HTTP는 stateless이다. 무상태, 즉 매 요청시 기본적으로 상태가 유지되지않음.
post - c
get - r
put - u (전체 덮어쓰기)
patch - update (일부 수정)
delete - d
etc method - head, options

----------------------------------------------------------------------------------------------

HTTP method 속성

- 안전성
get은 서버에 저장된 데이터를 변환하지 않음으로, 안전하다


- 멱등성 : 같은 요청에 대한 조회 결과가 동일한가?
post는 멱등성을 보장하지 않음 ( 생성이기 때문 )


- 캐시가능성 : 재사용을 위해 요청에 대한 응답을 저장할 수 있는가?
get, head, post 메소드는 캐시가 가능하나, 
일반적으로 get, head 정도만 캐시로 사용한다
( 변경 가능성이 적은 정적자원(HTML, CSS, IMAGE, JS...)을 주로 캐싱 )

캐시란? - 클라이언트가 서버에 한번 요청했던 데이터에 대해 매번 요청할 때마다 다시 전송할 필요 없도록 임시적으로 데이터 보관을 위한 장소
캐싱 - 이러한 행위

----------------------------------------------------------------------------------------------

HTTP 상태 코드 : HTTP 요청에 대한 처리 상태를 응답하는 코드
- 100~199 : 요청 수신 후 처리중인 상태, 잘 사용하지 않음
- 200~ : 정상 처리 완료, 200=성공/201=C성공/202=요청수신,데이터처리완료X( 랭킹 집계 )/204=요청성공, 응답 데이터X( delete )

- 300~ : 리다이렉션, 요청을 완료하려면 추가 행동이 필요한 상태 
ex) 카카오/구글 로그인 : 타 서비스 플랫폼의 location으로 갔다가 돌아와야함 (OAuth)

400~ : 클라이언트측 오류, 잘못된 문법 등으로 서버쪽에서 해당 요청을 수행불가
클라이언트의 요청이 잘못되었기 때문에, 같은 요청의 재시도는 실패한다
400=클라이언트가 HTTP요청 내용을 수정해서 보내야함
ex) get메서드로 만들어진 API인데 post로 보냈다던가..., API스택과 불일치, 데이터타입 불일치 등...
401=클라이언트가 리소스에 대한 인증이 필요함
ex) 로그인
403=서버가 요청을 받았지만 승인 거부, 주로 인가 관련 문제
ex) 일반 유저, 관리자의 권한 문제
404=요청한 리소스가 서버에 없다, 이를 이용하여 리소스를 숨겨놓기도 한다

- 500~ : 서버 에러, 클라이언트가 보낸 요청은 정상이지만, 서버가 처리하지 못함
재시도시 성공할 수도 있다 ( 서버가 다운된 상태였을 수도 있기 때문 )

500=대부분 500으로 처리함
503=서비스 이용 불가

100,200,300,400,500번대 - 즉 백단위의 의미만 알면 됨

API는 "애플리케이션 프로그래밍 인터페이스(Application Programming Interface)"의 약어로, 서로 다른 소프트웨어 애플리케이션이 정해진 규칙에 따라 데이터를 주고받으며 소통할 수 있도록 하는 방법

API : 프로그램끼리 대화하는 방법, 레스토랑의 메뉴판 - 요청과 응답 정의

API ( Application Programming Interface ), 즉 인터페이스(틀)다

각 API의 기능을 활용만 할 줄 알면 된다!
내부의 서버측 구현방식을 알 필요도 없고, 그것이 API의 장점이다

" 이 API를 사용하면, 이걸 쓸 수 있다, 이렇게 쓰면 된다 " 만 알면 충분.
마치 인터페이스처럼.



RESTful API : REST는 "어떻게 서버 구현을 알지 못하는 클라이언트가 API를 잘 이용할 수 있을지"에 대한 고민의 결과이다.
잘 설계된 REST API는 URL만 보고도 API의 역할을 예상 가능하다.

소통에 드는 리소스를 줄이기 위함

RESTful API 디자인 원칙 : 
1. 동사보단 명사를, 단수보단 복수를 
member->members

2. 마지막에 ' / ' 넣지 않기

3. _대신 - 사용, 대문자 사용하지 않기
/restful_SERVICES -> /restful-services

4. 확장자 포함하지 않기 (svg, png, exe 등... )
/image.svg -> /images

5. 계층화 하기
/items/{memberid}/members/{itemId} -> members/{memberid}//items/{itemId} 
맴버에 아이템이 있기 때문.

API는 인터페이스다!!!!

Lombok 라이브러리 : 보일러 플레이트 코드를 줄여주는 라이브러리, 자동으로 보일러 플레이트 코드를 생성하여 코드 가독성+유지보수성을 높힘

보일러 플레이트 코드 ( Boilerplate code ) 란?
getter/setter 메서드, 생성자, toString 메서드 등, 자주쓰고 반복적으로 작성되는 코드


주요 롬복 어노테이션 : 
1. @Getter/@Setter
2. @ToString
3. @NoArgsConstructor, @AllArgsConstructor, @RequiredArgsConstructor
- @NoArgsConstructor : 기본 생성자 생성
- @AllArgsConstructor : 모든 필드를 매개변수로 하는 생성자 생성
- @RequiredArgsConstructor : final 필드만을 매개변수로 하는 생성자 자동 생성
4. @Slf4j : 클래스에 로그를 남기기 위한 Logger객체를 자동 생성
ex) System.out.println(user);
-> log.info("user:{}", user); 
혹은 -> log.info(user.toString());

초기 웹페이지는 요청을 받으면 서버에 저장된 페이지를 보낼 뿐이었다 (정적, Static)

이러한 한계를 극복하기 위해, 요청이 들어왔을 때 서버에서 프로그램을 실행하여 동적으로 HTML을 만들어주는 방식이 생겨남 --- Servlet


Servlet : 서버에서 실행, 클라이언트의 요청을 처리하고 그 결과를 동적으로 생성하여 응답하는 자바 프로그램

- 자바 웹 어플리케이션에서 특정 URL 요청에 대한 비즈니스 로직을 실행

서블릿의 역할
1. 클라이언트가 보낸 데이터( HTTP Request )를 읽고 해석
2. 조회, 계산 등 비즈니스 로직을 수행
3. 결과를 생성하여 클라이언트에게 보냄
4. jakarta.servlet.http.HttpServlet 클래스를 상속받아 구현


서블릿 컨테이너 : 우리가 만든 서블릿 클래스( .java )는 스스로 실행될 수 없음
서블릿을 실행/관리하고 웹 서버와 통신하게 해주는 전문적인 환경이 필요, 이를 서블릿 컨테이너라고 함
대표적인 서블릿 컨테이너 - Tomcat

서블릿 컨테이너의 역할
1. 서블릿의 생명주기( 생성 - 실행 - 소멸 ) 관리
2. HTTP 요청을 받아 서블릿에게 전달
3. 서블릿의 응답을 받아 클라이언트에게 전송
4. 동시에 여러 요청이 들어와도 문제없이 처리


 서블릿 사용시 클라이언트-서버 요청&응답 흐름도
1. HTTP 요청이 서버에 들어옴
2. 서버는 Request와 Response 객체를 새로 만들어서 서블릿 객체를 호출
3. 서블릿 객체는 Request 객체에서 요청 정보를 꺼내서 비즈니스 로직을 수행
4. Response 객체에 응답 정보를 담아서 클라이언트에 응답을 전송
 
이 과정에서 개발자가 할 일
Request 객체의 요청 정보를 꺼내 비즈니스 로직을 수행, Response 객체를 만들기

API 명세서 : 클라이언트는 내부 구조를 모르는 채로 기능을 쓰기 때문에, 사용법이 적힌 설명서가 필요하다, 이 API를 알려주는 문서가 바로 API 명세서

손님 - 메뉴판(API명세서) - 서버 - 주방


API 명세서의 중요성 : 
1. 명확한 소통의 기준 : 소통시 오해를 줄여줄 수 있음
2. 프론트-백 동시 개발 가능 : 명세서라는 설계도가 있다면, 백엔드가 실제 기능을 만드는 동안 프론트엔드에서는 가짜 데이터로 화면을 미리 개발할 수 있다
3. 쉬운 유지보수 : 신입 개발자가 투입되어도, 명세서를 보면 시스템의 기능을 빠르게 파악 가능



API 명세서의 핵심 구성 요소
API 명세서에는 보통 다음과 같은 정보들이 포함

API 개요 : 이 API가 어떤 목적으로 만들어졌는지, 버전 정보, 전체 주소(Base URL) 등
Endpoint : API가 제공하는 개별 기능의 '주소' (URI)
Method : 해당 주소로 어떤 작업을 할지 결정 (HTTP Method)
Parameters : 요청할 때 꼭 보내야 하는 추가 정보들
Request Body : 생성하거나 수정할 데이터의 실체 (주로 JSON 형식)
Response : 요청 처리 후 서버가 돌려주는 응답 정보
Status Codes : 요청의 성공/실패 여부를 나타내는 숫자 코드

💬 API 문서의 형태는 굉장히 다양
다만, 백엔드 구현을 모르는 클라이언트 입장에서 API를 이용하는 것에 지장이 없도록 친절하고 자세하게 작성해주어야 함

서블릿의 문제점 : 몇가지 불편하고 반복적인 부분이 숨어있음
단순한 로직임에도, 코드가 복잡해짐

1. 낮은 응집도 : 
하나의 서블릿 클래스 내에 HTTP 요청 처리, 비즈니스 로직, 데이터 접근, 응답 생성 로직이 모두 혼재함. 이는 단일 클래스가 너무 많은 책임을 갖게 함.

2. 강한 결합도 : 
비즈니스 로직이 서블릿 API에 직접적으로 의존함. 이로 인해 웹 환경 밖에서 단위 테스트를 수행하기 매우 까다로움

3. 반복적인 상용구 코드 (Boilerplate Code) : 
요청 파라미터 파싱, 타입 변환, 응답 컨텐츠 타입 설정 등 모든 요청에 대해 반복적인 코드가 필수적으로 요구됨

우리가 서블릿에서 겪었던 가장 큰 문제는 역할 분담의 부재였다

이러한 문제점들을 해결하기 위해 관심사의 분리(SoC) 원칙에 입각하여 스프링 MVC를 사용하게 됨


MVC 패턴 : 
📚 MVC패턴이란?
MVC 패턴은 하나의 Servlet이 혼자 모든 것을 처리하던 문제를 해결하기 위해 
애플리케이션의 코드를 세 가지 역할로 명확하게 나누는 설계 방식
Model: 데이터와 비즈니스 로직을 담당 (예: 영화 데이터, 추천 로직)
View: 사용자에게 보여지는 화면(UI)을 담당 (예: HTML 페이지)
Controller: 사용자의 요청(Request)을 받아 Model과 View를 연결해주는 중간 다리 역할을 함

🤔 역할 분담이 왜 필요할까?
역할을 분리하면 코드가 훨씬 더 깔끔해지고, 유지보수와 테스트가 용이

📚 Controller
사용자의 HTTP 요청을 받아서 처리
URL 매핑을 통해 어떤 요청이 들어왔을 때 어떤 메소드를 실행할지 결정하고, 비즈니스 로직을 호출하고 데이터를 Model에 담아 View 이름을 반환
💡 @Controller 어노테이션이 붙은 클래스가 이 역할을 함
 
📚 Model
Controller에서 View로 데이터를 전달할 때 사용하는 객체
Key-Value 형태로 데이터를 저장하며, Controller에서 처리한 결과 데이터를 Model에 추가하면 View에서 해당 데이터를 참조할 수 있음 
주로 Model, ModelAndView 클래스를 사용
 
📚 View
사용자에게 보여지는 화면을 담당
 Model에 담긴 데이터를 사용해 HTML을 생성하고 클라이언트에게 응답
Controller가 반환한 View 이름을 ViewResolver가 실제 View 파일과 매핑시켜 렌더링
💡 Thymeleaf와 같은 템플릿 엔진이 이 역할을 함


✅ Thymeleaf
📌타임리프(Thymeleaf)란?
서버에서 HTML을 동적으로 만들어주는 자바 기반의 도구, 즉 템플릿 엔진.
→ HTML을 동적으로 만들어줄 수 있음에 주목
 
📚 타임리프(Thymeleaf)의 주된 역할
자바 코드(스프링 컨트롤러)가 보내주는 데이터를 실제 HTML 코드와 결합하여, 내용이 채워진 완성된 웹 페이지를 만드는 것
 
예를 들어, "${userName}님 환영합니다" 라는 HTML 템플릿이 있다면, 타임리프는 ${userName} 부분을 실제 사용자 이름인 "홍길동"으로 바꿔서 최종 페이지를 완성
 
🤔 그런데, HTML을 동적으로 표현할 수 있다는 뜻은 마치 프론트엔드 개발과 혼란을 일으킵니다.
백엔드(Thymeleaf)와 프론트엔드의 HTML 구현 방식에 어떤 차이가 있을까요?


✅ SSR과 CSR
📚 SSR(Server Side Rendering) - 서버에서 페이지를 랜더링
SSR은 서버가 웹 페이지에 필요한 모든 데이터를 채워 넣어 완벽하게 조립된 HTML 페이지를 브라우저에게 전달하는 방식
사용자는 거의 즉시 콘텐츠를 볼 수 있어 초기 로딩 속도가 빠르고, 모든 내용이 HTML에 포함되어 있어 검색 엔진 최적화(SEO)에 유리

타임리프가 바로 이 SSR 방식
💡 쉽게 말하자면 서버에서 완성된 HTML을 통째로 주는 것
 
 
📚 CSR(Client Side Rendering) - 클라이언트가 페이지를 직접 랜더링
CSR은 서버가 거의 텅 빈 HTML 뼈대와 자바스크립트 파일만 보내주는 방식
그러면 사용자 브라우저(클라이언트)가 자바스크립트를 실행해 필요한 데이터를 서버에 다시 요청하고, 브라우저가 직접 페이지를 조립하여 화면에 그림
💡 쉽게 말하자면 클라이언트에서 자바스크립트로 그 때 그 때 HTML을 그려내는 것
React나 Vue 같은 최신 프론트엔드 프레임워크들이 이 방식을 사용
 CSR은 초기 로딩은 느릴 수 있지만, 이후에는 앱처럼 부드럽고 빠른 화면 전환이 가능

 ✅ 1. 프론트 컨트롤러 패턴(Front Controller Pattern)
📌 프론트 컨트롤러 패턴이란?
모든 클라이언트 요청을 단일 진입점(Single Point of Entry)에서 처리하는 디자인 패턴
요청에 대한 공통 처리(보안, 로깅, 인코딩 등)를 중앙에서 효율적으로 관리할 수 있으며, 개별 요청을 처리할 핸들러(Controller)로 작업을 위임하는 역할
 
🤔 프론트 컨트롤러가 왜 필요할까?
💡 전통적인 방식의 문제점
각 서블릿마다 공통 로직(인증, 로깅, 인코딩 등)이 중복됨
새로운 기능 추가 시 모든 서블릿 수정 필요
관리가 어렵고 일관성 유지 어려움
 
 프론트 컨트롤러 패턴 해결책
💡 프론트 컨트롤러 패턴을 사용한다면?
모든 요청을 하나의 컨트롤러가 받아서 처리(공통 로직을 중앙화)
요청에 따라 적절한 컨트롤러로 위임
 
📚 프론트 컨트롤러 패턴의 장점!
공통 로직의 중앙화
일관된 처리 흐름
새로운 컨트롤러 추가 시에도 공통 로직 자동 적용
유지보수성 향상



✅ 2. 어댑터 패턴 (Adapter Pattern)
📌 어댑터 패턴이란 서로 다른 인터페이스를 가진 클래스들을 연결해주는 패턴
즉, 이는 다형성을 잘 활용한 패턴이라고 볼 수 있음

🤔 Spring MVC에서 왜 어댑터 패턴이 필요할까?
Spring에서는 다양한 형태의 컨트롤러를 지원하기 때문에, 프론트 컨트롤러 패턴만으로는 다양한 종류의 컨트롤러를 처리하기 어려움
이때 어댑터 패턴이 함께 사용되면 다형성을 활용해 쉽게 다양한 컨트롤 타입을 통일된 방식으로 다룰 수 있음
모든 컨트롤러 구현 방식이 동일한 interface를 implmenets 한다면, Controller를 모두 하나의 HandlerAdapter라는 타입으로 처리할 수 있게 됨


📚 어댑터 패턴의 장점!
✅ 다양한 컨트롤러 타입을 통일된 방식으로 처리(다형성)
✅ 새로운 컨트롤러 타입 추가 시 새 어댑터만 만들면 됨
✅ 기존 코드 수정 없이 확장 가능


✅ 3. DispatcherServlet
📌 DispatcherServlet이란?
DispatcherServlet은 프론트 컨트롤러 패턴(Front Controller Pattern)을 구현한 스프링 MVC의 핵심적인 프론트 컨트롤러
(Dispatcher Servlet은 프론트 컨트롤러 패턴과 어댑터 패턴을 기반으로 만들어졌다.)
 
 
스프링 MVC와 DispatcherServlet을 연관지어서 하나의 흐름도로 정리
① 요청
클라이언트가 웹 애플리케이션에 요청(Request)을 보내고, 이 요청은 가장 먼저 DispatcherServlet에 도달
② 핸들러 조회
DispatcherServlet은 HandlerMapping에게 요청을 처리할 Handler(=Controller)를 찾아달라고 요청
③ 핸들러 실행
DispatcherServlet은 HandlerMapping으로부터 받은 정보를 이용해 해당 Controller에게 요청 처리를 위임
④ ModelAndView 반환
Controller는 비즈니스 로직을 수행한 후, 결과 데이터(Model)와 뷰의 논리적 이름(View Name)을 담은 ModelAndView 객체를 DispatcherServlet에 반환
⑤ 뷰 해석
DispatcherServlet은 ModelAndView에서 뷰 이름을 추출하여 ViewResolver에게 전달하고, 해당하는 실제 View 객체를 찾아달라고 요청
⑥ 뷰 렌더링
DispatcherServlet은 ViewResolver로부터 받은 View 객체에게 모델 데이터를 전달하여 뷰를 렌더링하도록 요청
⑦ 응답
렌더링된 View의 결과물이 DispatcherServlet을 통해 클라이언트에게 최종적으로 응답(Response)으로 전달
