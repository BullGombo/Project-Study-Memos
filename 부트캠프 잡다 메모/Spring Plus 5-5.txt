분산 락


# ✅ 분산 락(Distributed Lock)은 무엇인가요?
 
분산 락은 여러 서버가 동시에 같은 작업을 수행하지 못하도록
“하나의 문을 만들고 한 명씩만 들어오도록 제한하는 장치”


비유하자면,
✔ 분산 락 = “입장 문지기(Doorman)”
 
서버 A가 먼저 작업을 시작하면
→ 문지기가 “지금 A가 처리 중. B는 잠시 대기.”라고 통제
서버 B는 문이 열릴 때까지 기다리거나, 실패 응답
작업이 끝나면 락을 해제하여 다음 요청이 들어올 수 있게 오픈


즉,
여러 서버(여러 출입문)가 있어도, 하나의 문으로 통합하여
한 번에 한 명만 들어오게 만드는 기술이 분산 락

---

## ✅ 서버 1대일 때는 낙관적 락 / 비관적 락으로 충분

서버가 1대일 때는 다음과 같은 구조

>
```
요청 → 단일 서버 → Hibernate/JPA → DB → 처리 완료
```

모든 요청이 서버 한 대에서 순차적으로 처리
Hibernate의 버전 방식(낙관적 락), DB의 Row Lock(비관적 락)이 정상적으로 동작
서버 내부에서 한 줄로 동작하기 때문에 동시성 문제가 크게 발생하지 않음
 
즉,
서버 1대 환경에서는 분산 락이 굳이 필요하지 않음



## ✅ 서버가 여러 대가 되는 순간 문제가 발생


서버가 두 대 이상이면 구조는 이렇게 바뀜


>
```
             ┌─ 서버 A → Hibernate → DB
클라이언트 →──┤
             └─ 서버 B → Hibernate → DB
```


✔ 서버 A와 서버 B는 서로의 상태를 전혀 알지 못함
메모리를 공유
Hibernate의 영속성 컨텍스트도 각각 따로 존재
어떤 서버에서 락이 걸렸는지 서로 알 수 없음
 
그 결과,
각 서버에서 동일한 로직이 동시에 실행될 수 있는 문제가 생김

--

## 비관적 락은 “DB 관여 순간”만 보호
## 분산 락은 서버 통제로 비즈니스 로직 전체를 보호

서버가 여러 대에서 다음을 모두 보호하려면 분산 락이 필요
- 잔액 조회
- 외부 결제 API
- 쿠폰 처리
- 로그 적재
 최종 UPDATE
#### 비관적 락은 이 중 마지막만 보호
 
분산 락은 전체 과정을 “한 사람씩 순서대로” 처리할 수 있도록 함
이미 처리중인 데이터가 있을 때 불필요한 메서드 실행 자체를 실행을 막을 수 있음


---

## 낙관적 락 / 비관적 락 / 분산 락 비교

| 종류                           | 관리 주체           | 보호 범위                 | 서버 여러 대에서 안전성   |
| ---------------------------- | --------------- | --------------------- | --------------- |
| **낙관적 락 (Optimistic Lock)**  | Hibernate (JPA) | UPDATE 시점의 충돌 감지      | ❌               |
| **비관적 락 (Pessimistic Lock)** | DB              | DB Row 수정 시점          | ⭕ (UPDATE 범위 내) |
| **분산 락 (Distributed Lock)**  | Redis 등 외부 시스템  | 로직 전체 (조회 → API → DB) | ⭕⭕⭕ (가장 안전)     |



이미지 스샷

> 순서
1. 서버 1, 서버 2 에서 동시에 DB 접근 허락 여부를 Redis에 문의
2. 서버 1이 먼저 들어와서 Redis에서 접근 여부를 생성해주고 Key를 생성해서 서버 1이 작업 중을 알림
3. 서버 2는 Redis로부터 서버 1이 작업중이라는 Key 의 존재 여부를 확인 (대기 or 재시도)
4. 서버 1 에서 key를 확보 했으니 DB에 접근
5. 작업이 완료되면 redis에서 생성한 Key를 회수


---

## ✅ Redis를 통한 분산 락

Redis를 이용한 분산 락 원리
Redis의 SETNX(Set if Not Exists) 명령을 이용해
“아직 없는 키만 등록” 할 수 있음

>
- 1. SETNX lock:account:1 요청 → 성공 시 락 획득
- 2. TTL(유효 시간)을 설정해 락이 자동으로 풀리도록 설정
- 3. 락을 얻은 서버가 안전하게 작업 수행
- 4. 작업 완료 후 DEL lock:account:1 으로 락 해제


>
락을 직접 구현할 때 고려할 점
- TTL 설정 : 서버 장애 시 락이 자동 해제되도록 필수 설정
- 락 주인 확인 : 락을 잡은 주체만 해제하도록 UUID 사용
- 재시도 로직 : 락 획득 실패 시 일정 시간 대기 후 재시도

# ✅ 락 획득시 전략

## 즉시 실패(Return Error) — “바로 실패 응답 보내기”
- 가장 단순하고 안전한 방식
 
언제 사용?
출금 / 주문 / 결제처럼 동일 요청을 중복 실행하면 안 되는 작업
이벤트 쿠폰 발급 등 선착순 로직


동작 방식
락이 이미 잡혀 있다면 : '이미 처리중이므로, 기다려주세요'


장점
Redis 부하 없음
중복 요청 방지 확실함

단점
사용자가 바로 실패 응답을 받을 수 있음



## 재시도(Retry) — “잠시 기다렸다가 다시 시도”
- 락을 얻을 때까지 잠시 대기→재시도 하는 방식
 
언제 사용?
조금 기다려서라도 반드시 성공해야 하는 작업
예: 상품 재고 감소, 장바구니 → 주문 변환 과정


장점
요청이 조금 늦어져도 성공 가능
 
단점
재시도를 너무 많이 하면 Redis에 부하
재시도 로직을 잘 설계해야 함 (백오프 필요)


---


✅ 분산 락의 단점 및 주의 사항
 
TTL과 작업 시간이 맞지 않으면 위험
TTL이 너무 짧으면 다음 문제가 발생합니다:
작업이 끝나지 않았는데 TTL이 먼저 만료 → 락 풀림
다른 요청이 들어와서 중복 실행 발생
 
Redis 장애 시 전체 동시성 제어가 무너짐
Redis가 다운되면:
락 잡기 불가능
동시성 제어 불가
요청이 모두 “동시에” 처리되는 상황 발생
 
락 남용 시 성능 저하
아무 로직에나 락을 걸면:
모든 요청이 순차적으로 처리되어 성능 저하
서버 스레드가 기다리다가 Timeout
Redis에도 부하
락은 “최소한 필요할 때만” 사용해야 합니다.
