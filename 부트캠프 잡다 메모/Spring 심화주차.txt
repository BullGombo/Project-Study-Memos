## JWT
JWT는 사원증이라고 생각하면 편하다.

### JWT 구조
Header.Payload.Signature
       Header        -     Payload    -       Signature
(암호화 알고리즘) - (암호화된 값) - (암호화시 사용한 Key 값)
Header는 암호화 타입과 알고리즘 (json형식)
Payload는 이름, 권한, id등의 정보 (json형식)
Signature는 고유한 인증번호같은 Key값

그래서 실제로 필요한건 Payload를 구성하는 정보지만, 암호화를 위해 헤더와 시그니처를 필요로하게 된다.

## Spring Security
Spring 기반 애플리케이션에서 인증/인가를 자동으로 처리하는 프레임워크

기본 설정이 세션 기반 인증으로 세팅되어있음

대표적으로 3가지 인증/인가 방식이 존재
세션,
jwt, 
OAuth2 인증 - 외부 서비스(구글, 카카오 등)를 통한 인증

JWT 기반 인증을 사용할 것이기 때문에, 
앞으로는 Spring Security를 커스터마이징해서 사용함

### Spring Security 프레임워크 주요 특징
- Filter 기반 보안 처리
- 인증/인가 로직을 개발자가 직접 구현하지 않아도 되는 구조 제공
- 다양한 인증 방식을 플로그인 형태로 확장 가능

build.gradle에 spring security 추가
implementation 'org.springframework.boot:spring-boot-starter-security'

Spring Security를 추가하면 Filter의 구조가 바뀜
FilterChain 내부에 SecurityFilterChain이 추가됨

#### Spring Security를 JWT기반으로 사용하기
Spring Security는 세션 방식이 디폴트, 따라서 JWT기반 인증을 사용하려면 
로그인 이후의 모든 요청을 JWT커스텀 필터에서 처리하도록 작업해야함
[사용자 요청]

# Filter, Interceptor, AOP

- 공통 관심사란?
Spring에서 웹 애플리케이션을 개발할 때, 수많은 요청마다 반복적으로 처리해야 하는 공통 로직이 존재
인증, 인가, 로깅, 트랜잭션, 예외 처리 등 여러 레이어에 반복적으로 등장하는 처리
핵심 로직과 분리하여 유지보수성을 높여야함

예를 들어:
모든 요청에 대해 로그를 남기고 싶다
로그인된 사용자만 특정 요청을 허용하고 싶다
요청이 들어올 때마다 요청 시간을 측정하고 싶다
특정 서비스 로직 실행 전후로 공통적인 검증이나 처리를 하고 싶다

이런 공통 기능을 모든 컨트롤러나 서비스 코드 안에 직접 작성한다면, 여러가지 문제점이 발생함


## 앞뒤 순서
Filter -> DispatcherServlet -> Interceptor -> Controller -> AOP -> Method
-> interceptor -> DispatcherServlet -> Filter ?

>
- Filter = 건물 입구 보안검색대
- Interceptor = 사무실 출입 리셉션
- AOP = 책상 위에 앉아서 문서 확인


[사용자 요청]
   ↓
[Filter] ← 요청 시작 시 로그 출력
   ↓
[Interceptor] ← Controller 실행 전/후 로그 출력
   ↓
[AOP] ← Service 메서드 실행 전/후/예외 시점 로그 출력


### ✅ Filter – 인증 토큰 처리 (ex. JWT)
- Authorization 헤더에 담긴 JWT 토큰 유효성 검사
ex) jwt ... [인증, 로깅, CORS 등]

왜 Filter에서?
모든 요청의 가장 앞단에서 작동
Spring DispatcherServlet 이전에 실행되어야 함
인증 실패 시 컨트롤러까지 가지 않도록 빠르게 차단 가능


### ✅ Interceptor – 로그인 여부 체크 (Controller 동작 전후)
- 로그인이 필요한 페이지 요청 시 세션 또는 인증 여부 검증
ex) 내가 작성한 개시글 삭제 등... [세션 체크, 권한 검사]

왜 Interceptor에서?
컨트롤러 실행 직전에 처리 가능
사용자별 세션, 권한 확인 로직 작성에 용이
요청 URL 기반으로 유연하게 경로별 적용 가능


### ✅ AOP – 실행 시간 측정 / 로깅 (Service로직 전후)
- Service 계층에서 비즈니스 로직 실행 시간 측정
ex) 시간 측정 ... [로깅, 트랜잭션, 예외처리 등]

왜 AOP에서?
특정 메서드 호출 전후에만 부가 기능을 넣고 싶을 때 유리
핵심 로직은 건드리지 않고 공통 처리만 분리 가능
다양한 메서드에 어노테이션으로 간편하게 확장 가능




💡
Spring Security에서 인가를 다 처리하지 않나요?
Spring Security를 사용하고 있다면 기본적인 인증/인가 처리는 이미 Spring Security의 필터 체인과 설정(authorizeHttpRequests, @PreAuthorize)에서 거의 다 담당하고 있어서 Interceptor의 사용 빈도가 확 줄어듭니다.
하지만 완전히 안 쓰는 건 아니고, 아래와 같은 경우엔 여전히 Interceptor가 유용하게 사용합니다.
작성자 본인 확인
ex) 내가 작성한 글만 삭제 가능한 경우
비즈니스 조건 
ex) 유료 회원만 접근
 
🚫 하지만 모든 것이 칼 같이 구분되지는 않습니다.
이건 무조건 AOP, 이건 무조건 Filter, 이건 무조건 Interceptor
이렇게 명확하게 구분되지 않고 애매한 상황이 있을 수도 있습니다.

# AOP

학습 목표
Aspect-Oriented Programming의 개념과 이점을 이해
@Aspect, @Around, @Before 등을 활용하여 횡단 관심사 로직을 분리


## AOP가 필요한 이유 : 관심사 분리

스크린샷 2025-12-02 180131

모든 메서드에 '시작 시간', '종료 시간'을 남기는 등

원하는 모든 메서드에 AOP로 중복 로직을 일괄 적용할 수 있음
AOP가 각 도메인의 메서드를 횡단하며 원하는 시점에 원하는 로직을 동작시킴

유지보수에 유리


### AOP 개념

- AOP에서 사용하는 핵심 용어
Aspect : 공통 관심사를 담는 모듈 (ex. 로깅, 트랜잭션, 권한 체크)
JoinPoint : 메서드 실행 지점
Advice : 실행 시점에 따라 작동하는 AOP 로직 (@Before, @After, @Around)
Pointcut : 어떤 JoinPoint에 Advice를 적용할지 정의 (표현식 기반)
Weaving : Advice를 적용하는 과정 (직접 Weaving을 다룰 일은 없음)


#### AOP 팁
어떤 것을 - 언제 - 어떻게
- 어떤 것을
PointCut -> 대상 지정

- 언제
Advice -> 실행 시점 지정

- 어떻게
Aspect에 메서드 정의 -> 어떻게 할 것인지 지정


### 주요 AOP 어노테이션 (Advice 개념)
@Aspect : Aspect 클래스임을 선언
@Around : 메서드 실행 전후 제어 (가장 유연)
@Before : 메서드 실행 전에 수행
@AfterReturning : 정상 반환 후 수행
@AfterThrowing : 예외 발생 시 수행
@After : 정상/예외 관계없이 실행 후 수행


#### 예시
@Around("execution(* com.advancestudy.user.service.UserService.*(..))")
위 AOP 선언의 
Aspect(횡단 관심사를 담은 클래스) ==> @Around 아래 전체
Advice(AOP 실행 시점) ==> @Around 로 정의
Pointcut(AOP 감지) ==> "execution(* com.advancestudy.user.service.UserService.*(..))"
JoinPoint(실행되는 메서드 지점) ==> com.advancestudy.user.service.UserService.*


#### AOP 활용 예시
로깅 : 모든 서비스 실행 시 로그 기록
트랜잭션 : 메서드 단위 트랜잭션 처리
권한 검사 : 관리자 권한 체크
예외 처리 : 공통 에러 메시지 처리
실행 시간 측정 : 성능 모니터링


### AOP 실제 동작 흐름 예시
     로그인 인증인가를 JWTFilter에서 처리
     user email 수정하는 메서드가 interceptor에서 처리됨 (아직 컨트롤러를 통과하지 않은 상태)
     userEmailUpdate라는 서비스단 메서드는 AOP가 수행됨

     필터 -> 인터셉터 -> AOP -> Controller 및 Service 메서드 -> AOP -> 필터
     요청이 들어옴 -> JWTFilter를 통과 -> WebMvcConfig-UserOwnerCheckInterceptor에서 설정된 URL에 해당 시, 인터셉터가 동작
     -> 인터셉터 통과 후 UserController에 접근 -> UserService의 Update메서드가 호출됨
     -> 호출된 Update메서드(서비스 레이어의 메서드)가 AOP 타겟으로 설정된 것을 확인 -> Update메서드 실행 전 -> TimeCheck AOP 동작
     -> Update메서드 실행 종료 -> TimeCheck AOP 동작
     -> AOP가 모두 끝나면 Controller로 return값을 넘겨줌
     -> JWTFilter를 통과하고 postman 등으로 결과 전달

반환 : 객체 혹은 객체의 필드
문법 : JPA 구현체에 의해 처리


## JPQL 사용 이유
✅ 객체 중심 개발에 자연스럽게 녹아들 수 있음
✅ 데이터베이스 독립성 확보
✅ 정적 타입 지원 + 자동 바인딩
✅ 복잡한 쿼리도 객체 기준으로 구성 가능


#### 흐름
쿼리메서드 -> 메서드를 JPQL로 변환 -> DB와 통신


## 지연로딩, N+1 문제

스샷 표 첨부

⚠️ 실무에서 주의할 점
EAGER는 불필요한 join 발생으로 성능에 악영향 줄 수 있음
특히 다대일, 일대다 관계가 많을수록 쿼리가 폭발할 수 있음
그래서 실무에서는 대부분 LAZY로 설정 + 필요한 경우만 직접 조회(fetch join, DTO 쿼리 등)


## ✅ N+1 문제란?
 
엔티티 1개를 조회했더니, 추가 쿼리가 N개 더 실행되는 현상
특히 연관된 데이터를 루프를 돌며 접근할 때 자주 발생합니다.

💡N+1은 항상 개발자가 방심하는 순간 발생하는 치명적인 독입니다.
 
좀 더 내용을 고도화 해서 예상하지 못하는 순간에 N+1이 발생하게 구성을 바꿔보겠습니다.


N+1 문제가 왜 심각한가?
단일 요청에서 수십 ~ 수백 개의 쿼리가 실행되며 DB에 불필요한 부하 발생
성능 저하로 이어져 트래픽이 많을수록 병목 가능성 증가
캐싱이 어렵고, 대량의 데이터 요청에서 응답 시간 급증
원인을 인지하지 못하면 문제 해결이 지연되거나 장애로 이어질 수 있음
💥 예를 들어...
회원 목록을 불러오고, 각 회원이 작성한 주문 목록을 보여줘야 하는데
회원 100명을 불러왔더니 주문 조회 쿼리가 100번 더 실행됨!
성능 테스트 시 문제가 되기 쉬운 패턴

# JPQL의 Fetch Join으로 N+1 해결


## Fetch Join

> 연관된 엔티티를 한 번의 쿼리로 함께 조회하기 위한 JPQL 키워드


🧨 기존 코드 (N+1 발생)

List<Post> posts = user.getPosts();
for (Post post : posts) {
	int commentCount = post.getComments().size();
}

쿼리 발생 (예시):
게시글 조회 1번
댓글 조회 쿼리 게시글 수만큼 (N회)


✅ Fetch Join 적용 코드

@Query("SELECT p FROM Post p JOIN FETCH p.comments WHERE p.user.username = :username")
List<Post> findAllWithCommentsByUsername(@Param("username") String username);


⚠️ Fetch Join의 주의사항

스샷


Fetch Join을 사용하면 중복 제거 작업을 직접 해줘야함 (Distinct)

# BatchSize로 N+1문제 해결

## BatchSize
- Hibernate 설정을 통해 컬렉션 또는 연관된 엔티티들을 배치로 로딩할 수 있게 해주는 기능
즉, LAZY 로딩이더라도 미리 설정된 개수만큼 한 번에 불러오게 해줌

JPA에서는 LAZY 로딩 전략이 기본이기 때문에 연관된 엔티티들을 개별 쿼리로 가져오며, 이로 인해 N+1 문제가 발생할 수 있음
BatchSize는 이러한 상황에서 Hibernate가 여러 엔티티를 한 번에 조회할 수 있도록 돕는 설정
즉, LAZY 전략을 유지하면서도 IN 절을 활용한 일괄 조회가 가능


### BatchSize 설정 방법

- 글로벌 설정 (application.yml)

spring:
	jpa:
		properties:
			hibernate:
				default_batch_fetch_size: 100


- 개별 설정 (엔티티 또는 컬렉션 필드)

    @OneToMany(mappedBy = "user", fetch = FetchType.LAZY) // ← 지연 로딩!
    @BatchSize(size = 100) 
    private List<Post> posts = new ArrayList<>();



✅ 주의 사항
 
BatchSize 크기 : 너무 작으면 효과 미미, 너무 크면 IN 쿼리가 길어져 성능 저하 가능
Fetch Join과 병행 : 사용 컬렉션 Fetch Join과 함께 쓰면 안 됨 (예: 페이징 깨짐)
LAZY 유지 조건 : 이 설정은 LAZY인 경우에만 유효함



## ✅ EntityGraph + DTO Projection

## EntityGraph를 활용한 N+1 해결
 
Spring Data JPA에서 지원하는 기능으로, 연관관계를 JPQL 없이 즉시 로딩하도록 지정 가능

@EntityGraph(attributePaths = {"posts"})
@Query("SELECT u FROM User u")
List<User> findAllWithPosts();

@EntityGraph(attributePaths = {"user", "comments"})
List<Post> findByUserUsername(String username);

장점 : JPQL 없이 선언적 방식으로 해결 가능 (+ 내부적으로 fetch join 실행)
단점 : 동적 제어 어려움, 복잡한 쿼리에는 한계

EntityGraph를 사용하면 쿼리메서드명을 마음대로 해선 안됨

# ✅ DTO Projection
## DTO Projection을 통한 최적화
- DB에서 원하는 컬럼만 선택하여 DTO로 직접 매핑하는 방식

예시 #1
@Query("SELECT new com.example.dto.UserDto(u.username, u.age) FROM User u")
List<UserDto> findAllUserDto();


예시 #2

    @Query("""
    SELECT new org.example.nbcam_addvanced_1.domain.post.model.dto.PostSummaryDto(
        p.content,
        SIZE(p.comments)
    )
    FROM Post p
    WHERE p.user.username = :username
    """)
    List<PostSummaryDto> findAllWithCommentsByUsername(String username);


장점 : 필요한 데이터만 조회, 불필요한 연관 로딩 방지
단점 : 엔티티가 아님 → 변경 감지 불가, 복잡한 로직 포함 어려움
💡DTO Projection을 활용하면 불필요한 Entity → DTO 변환 작업없이 바로 DTO 파일로 필요한 정보를 받을 수 있음

# 테스트 코드

## 테스트 코드의 필요성

> " 테스트 코드는 시간 낭비가 아니라, 미래의 나를 (야근으로부터) 지켜주는 보험이다. "

😵 테스트 코드가 없는 상황
기능 추가할 때마다 이전 코드가 깨질까 불안
수정 후 배포할 때마다 QA 팀의 확인을 기다려야 함
“어제 잘 되던 기능이 오늘 안 되는” 미스터리한 상황 발생

😎 테스트 코드가 있는 상황
기능 변경 후에도 테스트 한 번으로 안정성 확인 가능
코드 리팩토링 시 자신감 있게 변경 가능
새 기능을 추가해도 기존 기능이 깨지지 않음


#### ✅ 신입 개발자에게 테스트 코드가 특히 중요한 이유
 
신입으로 회사에 입사하면, 처음부터 새 프로젝트를 맡지 않습니다.
대부분 이미 수년간 운영되어 온 레거시 프로젝트를 담당하게 됩니다.
코드 수는 최소 수만 줄, 규모는 생각보다 훨씬 큽니다.
맡게 되는 업무는 대부분 이렇게 시작됩니다:
“이 기능 이름 좀 바꿔주세요.”
“여기 데이터 하나 추가해주세요.”
“이 부분 수정 부탁드려요.”
 
단순해 보이지만, 막상 수정해보면 오랜 시간이 걸립니다. 왜냐하면…
코드의 히스토리(변경 이력) 를 알지 못합니다.
수정한 코드가 어떤 기능에 영향을 주는지 범위를 파악하기 어렵습니다.
“이 정도면 괜찮겠지”라고 생각하고 배포하면 예기치 못한 버그가 발생할 수 있습니다.
 
그리고 팀원들이 일일이 여러분의 코드를 다 확인해줄 수도 없습니다. 😢
이때 최후의 보루가 테스트 코드입니다.
테스트 코드가 잘 작성된 프로젝트는, 신입 개발자가 인지하지 못한 오류를 배포 전에 미리 알려줍니다.
즉, 테스트 코드는 “개발자의 안전망”이자 “프로젝트 전체의 품질을 지켜주는 방패”입니다.
 
> ✔️ 테스트 코드는 나 자신을 지키는 것뿐만 아니라, 팀과 회사의 안정성까지 책임집니다.


회원가입 기능 수정 
-> 배포 후 오류 발생 (NullPointerException)
-> 배포 전 테스트 실패로 바로 수정

계산기 기능 리팩토링
-> 기존 로직 변경 시 결과 오류
-> 테스트 실행 시 “빨간불”로 즉시 감지

팀 프로젝트 협업
-> “이 코드가 진짜 동작하나요?” 질문 반복
-> 테스트 통과 = 동작 보장 (신뢰 형성)


# ✅ 테스트 피라미드
테스트에는 여러 “단계”가 존재하며, 단위 테스트(Unit Test) 가 가장 중요


        ▲
        │ E2E (전체 시나리오 테스트)
        │ Integration Test (통합 테스트)
        │ Unit Test (단위 테스트)
        └───────────────────────────
              빠르고, 자주, 자동화하기 쉬움


- Unit Test (단위 테스트): 하나의 클래스/메서드를 독립적으로 테스트
- Integration Test (통합 테스트): 여러 계층을 묶어 전체 플로우 검증
- E2E Test (종단 테스트): 실제 사용자 시나리오 기반 검증


## ✅ 테스트 코드가 주는 3가지 이점
1️⃣ 버그 조기 발견
버그는 “작성 직후” 잡을수록 고치기 쉽다.
테스트는 즉시 실행되므로 문제를 빠르게 확인 가능
QA 단계로 넘어가기 전 대부분의 에러 차단
 
2️⃣ 리팩토링 자신감 확보
테스트는 코드 리팩토링의 안전망이다.
기존 코드에 대한 신뢰 확보
“이전 기능이 깨지지 않았다”를 자동으로 증명
 
3️⃣ 협업 효율 극대화
테스트는 커뮤니케이션의 언어다.
코드 리뷰 시 “이 로직이 이런 동작을 해야 한다”를 코드로 보여줄 수 있음
새 팀원이 들어와도 “테스트 통과”를 기준으로 안정성 보장


### 테스트 예시

>
public class Calculator {
    public int add(int a, int b) {
        return a + b;
    }
}


테스트 코드 없이 수동으로 실행:

>
System.out.println(new Calculator().add(2, 3)); // 직접 눈으로 확인해야 함


테스트 코드로 자동 검증:

>
@Test
void add_두수를더하면_합이정상적으로계산된다() {
    Calculator calc = new Calculator();
    assertEquals(5, calc.add(2, 3));
}


- ✅ 실행 결과
초록불 ✅ → 정상 동작
빨간불 ❌ → 버그 발생
이제 테스트 결과가 코드 품질을 말해주는 시대



#### ✅ 테스트 작성의 실제 효과 비교
| 구분           | 테스트 없음        | 테스트 있음        |
| ------------ | ------------- | ------------- |
| **버그 탐지 속도** | 느림 (QA 이후 발견) | 빠름 (작성 직후 확인) |
| **리팩토링 안정성** | 불안함           | 자신감 있음        |
| **코드 품질**    | 낮음            | 지속적 개선 가능     |
| **협업 효율**    | 커뮤니케이션 부족     | 테스트로 의도 공유    |



#### ✅  정리
테스트 코드는 시간을 절약하고 신뢰를 높임
테스트는 “QA 팀의 일”이 아니라 “개발자의 자기 방어막” 
테스트를 작성한 개발자는,
→ 코드를 믿을 수 있고
→ 팀은 그 개발자를 믿을 수 있음

# JUnit

✅ JUnit이란?
 
> Java 진영의 대표적인 단위 테스트 프레임워크
- 테스트 코드를 작성하고 자동으로 실행·검증할 수 있도록 돕는 도구
- Spring Boot에 기본 포함 (spring-boot-starter-test 의존성 안에 JUnit 5 내장)
(springboot 3버전 기준)

JUnit 5 (Jupiter) : 
@BeforeEach, @AfterEach, @DisplayName 등 현대적 구조, Spring Boot 3+ 기본 탑재


## ✅ 주요 어노테이션

| 애너테이션          | 설명                                       |
| -------------- | ---------------------------------------- |
| `@Test`        | 테스트 메서드임을 표시                             |
| `@BeforeEach`  | 각 테스트 실행 **전** 실행 (초기화 용도)               |
| `@AfterEach`   | 각 테스트 실행 **후** 실행 (정리/clean up 용도)       |
| `@BeforeAll`   | 모든 테스트 시작 전 **딱 1번 실행** (⚠️ `static` 필요) |
| `@AfterAll`    | 모든 테스트 종료 후 **딱 1번 실행** (⚠️ `static` 필요) |
| `@DisplayName` | 테스트 이름 출력 시 보기 좋도록 커스텀                   |
| `@Disabled`    | 특정 테스트를 **임시 비활성화**                      |


### 기본 구조와 실행 흐름

```
import org.junit.jupiter.api.*;

class CalculatorTest {

    Calculator calculator;

    @BeforeEach
    void setUp() {
        calculator = new Calculator();
        System.out.println("📘 BeforeEach 실행");
    }

    @AfterEach
    void tearDown() {
        System.out.println("📕 AfterEach 실행");
    }

    @Test
    @DisplayName("더하기 기능 테스트")
    void add() {
        int result = calculator.add(2, 3);
        Assertions.assertEquals(5, result);
    }

    @Test
    @DisplayName("빼기 기능 테스트")
    void subtract() {
        int result = calculator.subtract(5, 2);
        Assertions.assertEquals(3, result);
    }
}
```

### 실행 순서

>
@BeforeEach → @Test → @AfterEach
@BeforeEach → @Test → @AfterEach


각 테스트 메서드는 독립적으로 실행됨
하나의 테스트 실패가 다른 테스트에 영향을 주지 않음



## ✅ 단언문(Assertions)
 
>
“테스트에서 기대하는 결과와 실제 결과를 비교하는 코드”

| 메서드                                         | 설명                                   |
| ------------------------------------------- | ------------------------------------ |
| `assertEquals(expected, actual)`            | 기대값과 실제값이 동일한지 확인                    |
| `assertTrue(condition)`                     | 조건이 **true**인지 확인                    |
| `assertFalse(condition)`                    | 조건이 **false**인지 확인                   |
| `assertThrows(Exception.class, () -> 실행코드)` | 특정 예외가 발생하는지 검증                      |
| `assertAll()`                               | 여러 개 검증을 **한번에 실행** (중간 실패에도 나머지 실행) |
| `assertNotNull(object)`                     | 객체가 **null이 아닌지** 확인                 |



### 예시

>
```
@Test
@DisplayName("예외 발생 테스트")
void divideByZero() {
    Assertions.assertThrows(ArithmeticException.class,
        () -> calculator.divide(10, 0));
}
```


### ✅ 테스트 이름 작성 팁
 
- 메서드명_상황_기대결과 규칙을 사용하면 읽기 쉬움


```Java
@Test
void add_2더하기3_결과는5() { ... }
```
 

#### 💬 “Given-When-Then” 패턴으로 시나리오 명확하게
Given: 사전 조건
When: 실행 동작
Then: 결과 검증

# 단위 테스트

단위(Unit): 프로그램의 가장 작은 실행 단위 (메서드, 클래스)
단위 테스트(Unit Test): 이 한 단위가 의도대로 동작하는지를 독립적으로 검증하는 것
💡 단위 테스트의 핵심은 “독립성”
단위 테스트는 빠르고 정확해야 함
하지만 DB, 외부 API, 파일 시스템 등에 의존하면 테스트 속도가 느려지고
실패 원인도 외부 요인과 섞여서 불명확해짐
→ 그래서 등장한 개념이 바로 Mock 객체(Mock Object)


## ✅ Mock 객체의 개념과 필요성

Mock 객체란?
실제 객체처럼 동작하지만, 테스트를 위해 가짜로 만든 객체
단위 테스트에서는 외부 의존성을 제거하기 위해 Mock 객체를 사용


### 📌 1. Mock이 필요한 이유

| 항목       | 실제 객체                     | Mock 객체                                  |
| -------- | ------------------------- | ---------------------------------------- |
| 실행 결과    | 실제 코드가 동작하며 DB/네트워크 접근 가능 | 설정된 값 또는 시나리오에 따라 시뮬레이션 실행               |
| 속도       | 상대적으로 느림 (환경 접근 포함)       | 매우 빠름 (외부 의존성 없음)                        |
| 제어 가능 여부 | 테스트 중 예측 불가 상황 발생 가능      | `when().thenReturn()` 등으로 테스트 결과를 완전히 제어 |
| 테스트 유형   | 통합 테스트(Integration Test)  | 단위 테스트(Unit Test)                        |
| 사용 목적    | 실제 환경 기반 검증               | 로직 검증 및 의존성 제거                           |


### 📌 2. 실제 객체 vs Mock 객체 비교


| 항목       | 실제 객체                     | Mock 객체                                  |
| -------- | ------------------------- | ---------------------------------------- |
| 실행 결과    | 실제 코드가 동작하며 DB/네트워크 접근 가능 | 설정된 값 또는 시나리오에 따라 시뮬레이션 실행               |
| 속도       | 상대적으로 느림 (환경 접근 포함)       | 매우 빠름 (외부 의존성 없음)                        |
| 제어 가능 여부 | 테스트 중 예측 불가 상황 발생 가능      | `when().thenReturn()` 등으로 테스트 결과를 완전히 제어 |
| 테스트 유형   | 통합 테스트(Integration Test)  | 단위 테스트(Unit Test)                        |
| 사용 목적    | 실제 환경 기반 검증               | 로직 검증 및 의존성 제거                           |



### 📌 3. Mockito 주요 어노테이션 및 기능

| 용어/어노테이션                              | 설명                              |
| ------------------------------------- | ------------------------------- |
| `@Mock`                               | Mock(가짜) 객체 생성                  |
| `@InjectMocks`                        | Mock 객체를 테스트 대상 클래스에 주입         |
| `@ExtendWith(MockitoExtension.class)` | JUnit5에서 Mockito 사용을 위한 확장 설정   |
| `when(...).thenReturn(...)`           | Mock 객체가 특정 상황에서 반환할 값을 설정      |
| `verify(..., times(n))`               | 특정 메서드가 호출되었는지, 호출 횟수는 몇 번인지 검증 |



### 📌 JwtUtilTest vs PostServiceTest 비교 요약
 
| 항목         | JwtUtilTest  | PostServiceTest          |
| ---------- | ------------ | ------------------------ |
| 외부 의존성     | 없음           | Repository 의존            |
| Mock 필요 여부 | ❌ 불필요        | ✅ 필수                     |
| 테스트 속도     | 매우 빠름        | 빠름                       |
| 실행 범위      | 순수 로직 테스트    | 비즈니스 로직 테스트              |
| 검증 포인트     | JWT 생성/검증 로직 | 데이터 처리 흐름 (save, find 등) |
| 목적         | 순수 단위 검증     | 의존성 분리 및 서비스 로직 검증       |
