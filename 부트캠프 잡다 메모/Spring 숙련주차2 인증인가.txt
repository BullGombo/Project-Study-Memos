## 인증(로그인) / 인가(권한 확인)

인증 - 회원정보 검증 및 로그인 후 SK/JWT발급
인가 - 클라이언트의 특정 요청(마이페이지 클릭 등)시 SK/JWT 확인

Session key 방식 / JWT 방식
- 사용자의 인증 상태를 서버에서 유지하기 위해 만들어진 장치


### 세션 ID 방식 - 서버에서 로그인 상태를 관리
session key(ID)를 발급, 전송/검증

클라이언트는 쿠키에 세션ID만을 가짐

서버가 세션 정보를 저장

장점 : 
서버에서 모든 세션 정보를 관리 - 보안에 유리함
쿠키에 세션 ID만 저장하므로 클라이언트에 민감한 정보가 남지 않음

단점 : 
사용자가 많아질수록 서버의 메모리나 DB에 부하가 커짐
확장성 문제가 존재

---

### JWT - Json Web Token 방식 - 클라이언트에서 로그인 상태를 관리
JWT를 발급, 전송/검증

( stateless )
서버가 클라이언트의 로그인 상태를 저장하지 않는 인증 방식
JWT 자체에 모든 인증 정보가 담겨있음

유저는 매 요청마다 서버에게 자신을 알려주게 됨


사용자의 정보/권한/만료 기한 등을 담은 암호화된 토큰(Access Token)을 생성하여 클라이언트에게 보내고, 서버는 이 토큰을 저장하지 않음
서버가 정보를 저장하지 않는다는 점이 중요!

클라이언트는 전달밭은 토큰을 로컬 스토리지나 쿠키에 저장함

클라이언트가 서버에 요청을 보낼 때마다 HTTP 헤더(쿠키 포함)에 토큰을 실어 보냄.
서버는 이 토큰의 서명을 검증하고 유효성을 확인하여 요청을 처리.


JWT 방식의 장점
1. 무상태 및 확장성 : 
- 서버가 토큰을 저장하지 않으므로 서버의 부하가 줄어들고, 여러 서버로 확장이 용이
2. 유연성 : 
- 웹 뿐만 아니라 모바일 앱 등 다양한 클라이언트 환경에서 사용 가능

JWT 방식의 단점
- 토큰이 탈취되면 만료시까지 악용될 수 있음
- 세션 ID보다 토큰의 용량 크기가 더 큼

## HTTP
무상태의 프로토콜 - Stateless
- 매 요청마다 HTTP요청 스스로가 자신이 누군지 밝혀야만 로그인 상태가 유지됨
그 방식중 대표적인 것이 Session Key와 JWT 방식

### Start Line
- 가장 첫 줄
- 요청 종류 (GET, POST... )
- 응답 성공 여부 (200 OK, 404 Not Found... )

### Header - Key : Value 형태 
- 보내는 사람, 받는 사람, 내용물의 종류 등 
메시지를 처리하는데 필요한 각종 설정/데이터
- 쿠키, Session Key와 JWT가 이곳에!!!!!
쿠키는 Key값이 Cookie(클라이언트-> 요청시) 혹은 Set-Cookie(서버-> 응답시)
#### Set-Cookie의 보안 설정 3가지
1. Secure-HTTPS통신일 때만 서버로 전송되도록 강제
2. HttpOnly-자바스크립트(document.cookie)가 쿠키에 접근하는 것을 막음
3. SameSite-다른 도메인에서 요청을 보낼 때 쿠키 전송 여부를 결정

## Session과 JWT

### Session
핵심 특징 : Stateful ( 상태 유지 ) - 서버는 클라이언트의 로그인 정보를 가지고 있음
1. 로그인을 하면 서버는 SessionKey를 클라이언트에게 발급, 로그인 정보를 저장
2. 이후 클라이언트의 매 요청 시 Session ID를 Header(쿠키)에 넣어 요청, 서버는 이를 로그인 정보와 대조하여 식별

### JWT
핵심 특징 : Stateless ( 무상태 ) - 서버는 클라이언트의 로그인 상태를 기억하지 않음
1. 로그인을 하면 서버는 JWT를 생성하여 클라이언트에게 반환
2. 이후, 클라이언트는 매 요청 시 JWT를 Header(쿠키)에 넣어 요청, 서버는 이를 식별

#### JWT의 구성요소
- aaaa.bbbb.cccc -> 점으로 구분된 세 부분으로 구성되어있음

Header(Json) - Payload(Json) - Signature

1. Header : 토큰 유형/서명 생성 알고리즘
2. Payload : 토큰에 담을 실제 정보
(즉 클레임: 사용자 정보 or 토큰 속성-만료시간, 발급자 등)
3. Signature : 

jwt Decoder로 세 파트를 조회가능 *마지막 파트는 보안화되어있음
Json객체를 Base-64Url 방식으로 인코딩 된 값을 볼 수 있음

서버는 클라이언트로부터 받은 JWT의 헤더+페이로드+비밀키로 다시 Signature를 만들어 받은 Signature와 대조하여 위변조 여부를 판단
